# AE（不良事件）漏报风险核查技能

基于EDC导出的全量Data Listing，自动识别临床试验中实验室检查、生命体征、心电图等指标的基线-基线后异常变化，匹配CTCAE分级标准，评估AE漏报风险并生成交互式核查报告。

## 触发条件

当用户消息中包含以下任一关键词时，激活此技能：
- "AE漏报"
- "实验室异常"
- "实验室检查异常"
- "AE漏报风险"

## 使用前须知

此技能是一个**对话式工作流**。Agent需要在每个阶段向用户询问必要信息，不能跳过或臆断。工作流如下：

---

## 阶段1：收集必要文件

激活后，若用户仅输入触发词而未上传任何文件，Agent应提示用户上传以下文件：

### 必需文件

1. **全量Data Listing（Excel格式，.xlsx）**
   - EDC系统导出的原始数据表（如FormExcelAllVersion格式）
   - 必须是全量数据，不可仅包含增量数据
   
2. **CTCAE分级文件（PDF格式）**
   - CTCAE v5.0 或 v6.0
   - 中英文版均可
   - 需提醒用户确认是哪个版本号（如5.0、6.0）

### 可选文件

3. **受试者分组表（Excel格式，.xlsx）**
   - 包含各受试者所在治疗组信息
   - 若用户确认无此资料，后续报告将不展示分组信息

### 提示话术模板

```
要开始AE漏报风险核查，请上传以下文件：
1. 全量Data Listing（EDC导出的Excel文件，如FormExcelAllVersion格式）
2. CTCAE分级文件（PDF，请确认版本号：5.0还是6.0？）
3. 受试者分组表（可选，Excel格式。如无此资料，报告将不展示分组信息）

请上传文件后，我会继续引导确认其他必要信息。
```

## 阶段2：确认构建模式

文件上传后，询问用户构建模式：

### 模式A：首次全量构建
- 不需要既往报告
- 从Data Listing中全新解析所有数据
- 生成完整报告

### 模式B：增量构建（基于既往报告的更新）
- 需要上传既往的HTML报告
- 需要上传最新版的全量Data Listing（不是仅包含增量数据的listing）
- 自动比对既往报告与新数据，标记新增/更新的数据点

对于CTCAE分级中不涉及的指标（如低密度脂蛋白胆固醇、某些尿常规项目等），需用户确认"异常"判定的阈值：

```
对于CTCAE分级中不涉及的指标（如某些生化/尿常规项目），

```

## 阶段4：解析Data Listing

Agent需要自动解析用户上传的Data Listing。解析规则如下：

### 4.1 识别相关Sheet

在Excel的所有sheet中，自动识别以下类型的sheet：

| 数据类型 | 识别关键词 | 关键列 |
|---------|-----------|--------|
| 受试者访视(SV) | "SV"、"访视" | 受试者编号、访视名称/日期 |
| 人口学(DM) | "DM"、"人口" | 受试者编号、年龄、性别 |
| 既往病史(MH) | "MH"、"病史"、"既往" | 受试者编号、疾病名称、开始/结束日期 |
| 不良事件(AE) | "AE"、"不良事件" | 受试者编号、AE名称、开始日期、严重程度 |
| 合并用药(CM) | "CM"、"合并用药" | 受试者编号、药物名称、开始日期 |
| 实验室-血常规 | "LB_HEM"、"血常规"、"HEM" | 受试者编号、采样日期、指标名称、结果、下限、上限、单位、临床评估 |
| 实验室-血生化 | "LB_CHEM"、"血生化"、"CHEM" | 受试者编号、采样日期、指标名称、结果、下限、上限、单位、临床评估 |
| 实验室-尿常规 | "LB_URI"、"尿常规"、"URI" | 受试者编号、采样日期、指标名称、结果、下限、上限、单位、临床评估 |
| 实验室-其他 | "LB_" | 受试者编号、采样日期、指标名称、结果、下限、上限、単位、临床评估 |
| 生命体征(VS) | "VS"、"生命体征"、"VSW" | 受试者编号、检查日期、检查项目、结果、单位、临床评估 |
| 心电图(EG) | "EG"、"心电图"、"ECG" | 受试者编号、检查日期、QTcF/心率/QRS等、临床评估 |
| 身高体重(HW) | "HW"、"身高体重"、"体格" | 受试者编号、测量日期、体重/身高 |

**关键原则：不要硬编码sheet名称。** 应基于sheet名称的关键词模糊匹配和列名模式识别来自动判断每个sheet的数据类型。

### 4.2 解析各Sheet的数据字段

对每个识别的sheet，自动提取以下维度的数据：

1. **受试者标识**：受试者编号、研究中心/中心名称
2. **时间信息**：访视名称、访视日期、访视点（如V1/V2/V3或"访视1/2/3"或"D7±1"等）
3. **检测指标**：指标名称、检测结果（数值）
4. **参考范围**：正常值下限、正常值上限
5. **临床评估**：临床意义评价（正常/异常无临床意义/异常有临床意义）
6. **关联信息**：异常说明、关联的AE编号、关联的MH编号、动态链接

### 4.3 临床意义解释列检测（强制步骤，不可跳过）

**规则：** CS（异常有临床意义）记录**必须**有临床意义解释。解析完成后，Agent**必须**执行以下验证，**在进入风险分析之前**向用户报告结果并获得确认。

**步骤1：统计CS记录的解释覆盖情况**

自动扫描所有CS记录，检查以下三个列的填充情况：
- 「动态链接」列 — 常见格式：
  - `MH+4+血小板减少`（加号分隔，含序号和术语）
  - `AE+1+血胆红素增高`（同上）
  - `AE：xxx` / `MH：xxx`（冒号分隔）
- 「异常说明」列（如"若异常有临床意义，请说明"、"VSABCS"等）
- 「备注」列

**步骤2：向用户呈现验证结果**

```
CS解释验证结果：
  CS记录总数: N 条
  有动态链接: N 条（格式: MH+N+术语 / AE+N+术语）
  有备注: N 条
  有异常说明: N 条
  三者全无: N 条 ← 若>0，必须暂停并询问用户
```

**步骤3：条件性暂停**

- 若「三者全无」> 0：**必须暂停**，询问用户确认临床意义解释列的位置
- 若所有CS记录均有解释：输出验证通过摘要，继续执行
- **禁止**在未完成CS验证的情况下进入风险分析阶段

### 4.3b 访视列识别与V{n}提取

EDC导出的Data Listing中，访视信息可能存在于以下任一列（按优先级）：
1. 「数据节」列 — 值如"双盲治疗期V2（D1）"、"筛选/导入期V1（D-7~D-1）"
2. 「访视」列 — 标准访视名称
3. 「访视点」/「Instance顺序号」列

Agent应自动检测上述列，不可仅查找"访视"列。

**V{n}提取规则**：从访视名称（数据节/访视列）中提取"V1"、"V2"等编号。
- "筛选/导入期V1（D-7~D-1）" → 提取 V1
- "双盲治疗期V2（D1）" → 提取 V2
- 若名称中无"V{n}"，不提取（不默认填"V1"）

显示规则：仅当V{n}不在压缩访视名中时才显示为子标签（如"D1"下方显示"V2"，但"筛选V1"不重复显示"V1"）。

### 4.4 排除只有筛选期/基线期的指标

某个指标如果所有受试者都**仅**在筛选期或基线期有检测数据（基线后无任何数据），则该指标不纳入分析。

### 4.5 识别缺失数据的提醒

解析过程中若发现以下情况，需提醒用户确认：
- 未发现生命体征(VS)相关sheet
- 未发现心电图(EG)相关sheet
- 实验室sheet中缺少参考范围（下限/上限）列
- 数据中有"访视名称"列但缺少"访视日期"（或反之）

## 阶段6：执行核心风险分析

### 6.0 排除规则（新增）

以下情形不纳入风险分析：

1. **仅有一次检测**：某指标在整个研究中仅有一次检测记录（所有受试者总计仅1条），不纳入。
2. **基线异常→基线后恢复正常**：基线已异常，但基线后恢复至正常范围内，且无CTCAE升级，不列为风险项（反之基线正常→基线后异常才是目标场景）。

### 6.1 CTCAE匹配

使用CTCAE PDF文件，执行以下匹配：

1. 确认CTCAE版本号（5.0 / 6.0）
2. 对Data Listing中识别出的每个检测指标，在CTCAE PDF中搜索匹配的条目
3. 提取各级别（1-5级）的判定标准和阈值

**CTCAE匹配映射规则（示例，非穷尽）：**

| 检测指标 | CTCAE匹配条目 | 匹配逻辑 |
|---------|-------------|---------|
| 白细胞计数 | 白细胞计数降低/升高 | 指标名称直接匹配 |
| 中性粒细胞计数 | 中性粒细胞计数降低 | 指标名称直接匹配 |
| 血红蛋白 | 贫血 | 关键词关联匹配 |
| 血小板计数 | 血小板计数降低 | 指标名称直接匹配 |
| ALT/AST | 丙氨酸氨基转移酶升高/天门冬氨酸氨基转移酶升高 | 缩写匹配 |
| 总胆固醇 | 高胆固醇血症 | 关键词关联匹配 |
| 甘油三酯 | 高甘油三酯血症 | 关键词关联匹配 |
| 低密度脂蛋白胆固醇 | ⚠️ 不应匹配"高胆固醇血症" | 需要专门的LDL条目 |
| 肌酐 | 肌酐升高 | 指标名称匹配 |
| 血葡萄糖 | 高血糖症/低血糖症 | 关键词关联匹配 |
| 体重 | 体重增加/体重降低 | 通过体重变化百分比匹配 |
| QTcF间期 | QT间期延长 | 缩写匹配 |
| 收缩压 | 高血压 | 关键词关联匹配 |
| 舒张压 | 高血压 | 关键词关联匹配 |

**匹配原则：**
- 优先精确匹配（指标名称完全一致）
- 其次缩写匹配（ALT→丙氨酸氨基转移酶）
- 再次关键词关联匹配（血红蛋白→贫血）
- ⚠️ 注意避免错误匹配：如低密度脂蛋白胆固醇不匹配为"高胆固醇血症"
- 对于无法匹配CTCAE的指标，标记为"无CTCAE分级"

### 6.2 异常筛选流程

对每个检测指标，按以下流程筛选异常：

#### 步骤1：基线-基线后异常筛选

(a) **基线正常 + 基线后异常**：基线值在正常范围内，且基线后有任一访视的检测值超出正常范围（高于上限或低于下限）

(b) **基线异常 + 基线后CTCAE分级上升**：基线值已异常，且基线后有任一访视的CTCAE分级较基线上升（如从1级升至2级）

(c) **基线异常 + 基线后无CTCAE匹配的指标变化≥阈值**：对于CTCAE中无法匹配的指标，基线值已异常，且基线后较基线变化≥用户指定的阈值（默认20%）

#### 步骤2：风险分级

对经过步骤1筛选的记录，进行风险分级：

① **高风险**（需要重点核查）：
- "临床意义评价"为"异常无临床意义"(NCS) 或 未评价 的异常记录
- 且无可合理解释异常的AE/MH记录

② **关注/中风险**（需补充说明或确认）：
- AE/MH记录不足以充分解释异常程度
- 或同一受试者同一指标中NCS与CS评价不一致
- 或所有异常均为CS但无AE/MH解释

③ **低风险**（已有合理解释）：
- 研究者评估为"正常"但技术性超出参考范围（如BASO从0.01→0.02）
- 已关联AE/MH且该AE/MH可合理解释异常
- 同一受试者同一指标所有异常均判定为CS且说明一致

### 6.3 AE/MH关联判断规则

当Data Listing的"临床意义解释"/"异常说明"列中引用了AE或MH编号时：

1. 提取AE/MH编号（如"AE1"、"MH3"等）
2. 查找对应的AE/MH记录，获取其名称/术语
3. 判断该AE/MH是否能合理解释当前指标异常：

**可解释的关联（示例规则，需基于实际AE/MH内容动态判断）：**
- 血常规指标异常 + 感染/AE（如白细胞升高+咽喉炎AE）
- 血脂指标异常 + 高脂血症AE/MH
- 肝功能异常 + 肝病/脂肪肝相关AE/MH
- 贫血相关指标 + 贫血MH
- 体重异常 + 体重相关AE/MH（如体重增加AE、食欲减退AE）

**不可解释的关联（示例规则）：**
- 血白细胞升高 + "尿白细胞升高"MH（不同样本类型）
- 任何血液指标 + 尿液相关异常（不同检测体系）
- 指标方向不匹配（如体重增加 + 体重减轻AE）

**判断逻辑：**
- 使用关键词模糊匹配判断AE/MH术语与检测指标的相关性
- 对于体重：区分增加/降低方向，仅匹配方向一致的AE/MH
- 对于尿液检测指标：不匹配血液系统的AE/MH
- 对于血液检测指标：不匹配尿液系统的异常

## 阶段7：生成交互式HTML报告（日常医学监察用）

### 7.1 报告结构

报告为单个HTML文件，包含以下结构：

#### 头部
- 报告标题："实验室检查/生命体征异常AE漏报风险核查报告"
- 项目信息：数据来源、生成日期
- CTCAE版本号、覆盖率统计

#### 中心Tab切换
- 按研究中心（site）分Tab展示
- 每个Tab下包含该中心的数据

#### 汇总面板
- 高风险数量、关注项数量、低风险数量、异常受试者数、总项次

#### 筛选项
- 按受试者编号筛选、按风险级别筛选、快速清除筛选

#### 指标导航
- 按检测类别分组（血常规/血生化、尿常规、生命体征、心电图、体重等）
- 点击跳转至对应指标详情

#### 指标详情区
- 按高→中→低风险排序
- 每个受试者-指标组合展示为卡片：
  - 受试者编号 / 指标名称
  - 风险标签（高风险/关注/低风险）
  - 异常计数、CTCAE升级计数标签
  - AE/MH解释状态标签
  - 相关AE/MH信息面板（含严重程度/与药物关系）
  - 基线信息
  - 完整检测数据表格（10列）：
    - 访视（含V{n}子标签，从数据节/访视名称中提取，不存在则不显示）、日期、检测结果、单位、参考范围、方向、CTCAE分级、临床意义、临床意义解释、较基线变化
  - 基线数据行：浅蓝色背景（#e3f2fd）

#### 增量构建特有标记
- 首页：按中心/个例汇总新增的各指标风险
- 数据点标记「较前次更新」标识
- 完整报告包含所有数据（原数据+增量数据）

### 7.2 美学规范

- 配色：主色#FF9900（橙色），背景#f4f6f8（浅灰），深色标题#1a1a2e
- 字体：系统默认（-apple-system, BlinkMacSystemFont, sans-serif）
- 卡片阴影：轻薄（box-shadow: 0 1px 3px rgba(0,0,0,.05)）
- 表格：紧凑型，斑马纹可选
- 响应式：max-width 1600px，适合大屏查看
- 高风险卡片：红色左边框 (#e74c3c)
- 中风险卡片：黄色左边框 (#FF9900)
- 低风险卡片：绿色左边框 (#27ae60)
- 异常高值：红色文字显示
- 异常低值：蓝色文字显示
- 基线行：浅灰底色

### 7.2b 显示逻辑规则（防Bug清单）

以下规则源于已知缺陷，编写或修改 `generate_report.py` 时**必须遵守**：

**规则1：CS/NCS判定必须先检查NCS**

临床评估值"异常无临床意义"包含"临床意义"子串。若先检查 `"临床意义" in cs_raw`，NCS会被错误匹配为CS。

```python
# ✗ 错误：NCS会被第一个条件命中
if "临床意义" in cs_raw:
    cs_html = "CS"
elif "无临床" in cs_raw:
    cs_html = "NCS"

# ✓ 正确：先检查NCS（更具体的条件优先）
if "无临床意义" in cs_raw or "NCS" in cs_raw.upper():
    cs_html = "NCS"
elif "有临床意义" in cs_raw or cs_raw.strip().upper() == "CS":
    cs_html = "CS"
```

**规则2：访视名称缩写必须保留主标签**

`compact_visit()` 提取括号内容时，"提前退出（±3d）"会只取"±3d"，丢失"提前退出"。

```python
# ✓ 正确：对特殊访视做前置处理
if "提前退出" in s:
    return "提前退出"
if "计划外" in s:
    return "计划外访视"
# 然后再提取括号内容
```

**规则3：动态链接正则必须支持加号分隔格式**

EDC导出的动态链接格式为 `MH+4+术语` 或 `AE+1+术语`（加号分隔），非 `MH4` 格式。

```python
# ✓ 正确：两种格式都支持
re.finditer(r"(AE|MH)\+(\d+)\+", desc)  # 加号分隔
re.finditer(r"(AE|MH)\s*(\d+)", desc)    # 直接/空格分隔
```

### 7.3 风险回复建议模板

对每条异常，自动生成核查判断与回复建议：

**高风险/无AE解释：**
```
未见可解释该异常的AE/MH/CM记录。建议确认是否漏报AE，或补充研究者医学判断依据。
若确认漏记AE/MH，应补录相应AE/MH记录，补充起止日期、严重程度/CTCAE分级、
与研究药物关系、处理措施及转归；若不作为AE，应由研究者在源文件/EDC说明不构成AE的医学理由。
```

**中风险/AE解释不足：**
```
临床意义解释中引用的AE/MH与该指标异常关联不足。建议研究者补充说明为何该AE/MH足以解释
本项异常。若不能解释，应评估是否存在需补记AE的独立事件。
```

**低风险/有合理解释：**
```
该异常已在临床意义解释或AE/MH记录中得到对应说明，暂不作为AE漏报高风险。
建议核查时保留原始说明与AE/MH编号的交叉引用。
```

## 执行总则

### 数据解析通用原则

1. **不硬编码**：不将前期项目的指标名称、sheet名称、访视名称等写入代码。所有数据元素应基于实际上传文件的解构结果动态判定。
2. **自适应**：自动识别Excel的sheet结构、列名映射。不同EDC系统导出的格式可能不同，需要灵活适配。
3. **透明**：每一步解析结果应向用户展示摘要，让用户确认是否正确识别。
4. **容错**：部分数据缺失不应阻断整体流程。缺失部分应标记并提醒用户。

### 排除项（相较于前期RUX-03-002项目）

1. ❌ 去除"对于18岁以下人群，应评估体重增加是否符合生长发育正常曲线标准"的要求
2. ❌ 去除JAK类风险标记的所有逻辑和提示

### 输出目录

除非用户另行指定，所有产物默认保存至用户指定的工作目录。首次运行时询问用户期望的输出路径。

### 中间数据保存

每次生成报告时，同时保存一份JSON格式的中间数据文件，包含：
- 解析后的所有检测记录
- 风险判定结果
- AE/MH关联信息
- 元数据（生成时间、数据来源、CTCAE版本）

该JSON文件供后续增量构建使用。

## 关键脚本和资源

### scripts/parse_listing.py
Data Listing解析器。输入Excel文件路径，输出结构化的JSON数据。自动识别sheet类型、提取各维度数据。

CTCAE PDF解析器。输入PDF文件路径和版本号，输出各检测指标的CTCAE分级标准字典。

HTML报告生成器。输入分析结果JSON，输出交互式HTML报告。支持全量和增量两种模式。

### scripts/main.py
主流程编排脚本。串联上述各模块，处理用户交互和参数传递。
